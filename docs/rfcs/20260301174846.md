# 20260301174846 — v1.1.0: Windows Registry Fix & Installer Overhaul

## Request

Release v1.1.0 of Yomitan API.

The primary bug: on Windows 11, the installer did not correctly write the registry key that browsers use to locate the native messaging host manifest. Running as administrator did not help because the correct fix was in the registry write logic itself, not permissions. This made Yomitan API non-functional on Windows for most browsers.

## Implementation Details

### Root Cause Analysis

**Problem:** After running `install_yomitan_api.py` on Windows, the browser could not find the native messaging host because:

1. **Missing Chromium registry path** — `PLATFORM_DATA["windows"]["manifest_install_data"]["chromium"]` had no `registry_path` key, so `winreg.CreateKey` received an invalid argument and crashed silently.
2. **Unclosed registry key** — `winreg.OpenKey()` was called without a `with` context manager. On failure or exception, the handle would leak. The bare `winreg.CloseKey()` at the end was never reached if any exception occurred.
3. **Multi-browser manifest overwrite** — when installing for "all browsers" (now the default), each browser's manifest was written to the same file `yomitan_api.json`, causing each iteration to overwrite the previous. The registry key for each browser pointed to a manifest that may have belonged to a different browser (different `allowed_origins` / `allowed_extensions` key).
4. **Windows Store Python (UWP)** — Python installed from the Microsoft Store operates in a sandbox. It fails to write the necessary registry keys for the native messaging host even when run as administrator. This makes it **unsuitable for setup** (`install_yomitan_api.py`), though it remains **suitable for running** the API server (`yomitan_api.py`) once the registry is configured.

### Decisions
...
- **Added Runtime Warnings**: The installer now detects if it is running under Windows Store Python and warns the user that it is unsuitable for setup due to registry limitations, while noting it is fine for running the API later.

- **Per-browser JSON filenames on Windows**: `yomitan_api_{browser}.json` instead of `yomitan_api.json`. Non-breaking for Linux/macOS which use a single shared file.
- **Registry key uses `with` context manager**: guarantees `CloseKey` is called even on exception.
- **Try/except per browser**: failure for one browser does not abort installation for others.
- **Default to install all browsers**: option `0` (all) is now the default. Users can still pick a specific browser.
- **Platform alias matching changed** from `==` to `startswith()`: handles edge cases like `freebsd` variants more robustly.

### Other Improvements (same release)

- **`yomitan_api.py` server robustness**:
  - Handles missing `Content-Length` header gracefully (was a crash).
  - Checks `yomitan_response` for `None` and missing keys before indexing (was a crash when the extension was not yet ready).
  - Full `try/except` in `do_POST` returns a `500` response instead of letting the exception propagate.
  - `delete_crowbarfile()` wrapped in `try/if exists` — no longer crashes if the file is missing.
  - `get_message()` guards against reading more than 33 MB (Python 3.9 Windows limitation).
  - `allow_reuse_address = True` — prevents "address already in use" errors on quick restart.
  - `delete_crowbarfile()` moved to `finally` block — guaranteed to run on any exit.
  - Server startup wait reduced from 5 s → 2 s.
- **Type annotations** added throughout (`from typing import Any, Optional`).
- **Code style**: reformatted with `ruff`, long lines split, `os.path.join` used consistently.
